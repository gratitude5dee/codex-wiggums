# Operating Rules

Use these rules when Ralph is active.

1. Search before editing or claiming context.
2. Restate the objective as something observable, not aspirational.
3. Make the `done_when` clause specific enough that another engineer could verify it.
4. Prefer a narrow test or check before a broad one when the task is code-facing.
5. Keep each iteration scoped to one concrete improvement or blocker.
6. Use `<promise>COMPLETE</promise>` only when the work satisfies the stated checks.
7. If the task is blocked, report the blocker concretely instead of claiming progress.
8. Use `peek` when you want the current wrapped prompt without spending an iteration.
9. Use `cancel` to stop stale loops and keep the archive as a history of intent.
